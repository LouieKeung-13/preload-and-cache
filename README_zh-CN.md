将预先准备好的研究报告、知识文档或分析材料一次性注入语言模型的上下文窗口，然后通过**上下文缓存**在后续多轮对话中重复使用，从而消除对同一份静态知识的冗余 token 消耗。

> **一句话概括：** "读一次书，之后反复提问都不用重读。"
> 
> 它**不是**用来"搜索互联网并总结"的方案。

---

## 核心特性

| 特性 | 说明 |
|------|------|
| **一次性注入** | 50KB+ 长文档直接读入上下文，后续对话零写入 |
| **高缓存命中率** | 注入后后续轮次命中率达 80–97% |
| **零部署成本** | 无需额外服务，直接利用 OpenClaw 内置缓存 |
| **互补流水线** | 与 Dynamic Token Compression 形成完整上下文管理策略 |

## 适用场景

### ✅ 适合

| 场景 | 示例 |
|------|------|
| 📊 研究分析 | 注入 50KB+ 报告，然后从多个维度追问 |
| 📚 知识库问答 | 注入产品文档/合规手册，然后反复查询 |
| 📝 报告审阅 | 注入完整提案，然后逐段讨论 |
| 🧠 深度调研 | 注入行业数据，然后从多角度分析 |

### ❌ 不适合

| 场景 | 原因 |
|------|------|
| 🌐 实时数据采集 | 需要动态检索；缓存上下文无法反映新内容 |
| 💬 日常聊天 | 注入成本远超单轮对话收益 |
| 🔄 频繁切换文档 | 每次切换都需要重新注入，浪费缓存 |
| 🔍 网页搜索 + 总结 | 属于不同的技术栈（如 MCP + 实时压缩） |

## ⚠️ 前置条件与基线考量

### 典型使用场景下的基线缓存命中率

在生产环境中，系统提示词包含动态更新的 artifacts（包括 memory 文件、skill 配置、心跳例行程序和子 agent 启动开销），普通对话轮次的基线缓存命中率通常在 **20–30%** 范围内。这个数字反映了因例行操作修改提示词表面而导致的持续上下文漂移的综合影响。

### 注入静态文档后的预期提升

当注入一份 substantial 的静态文档（≥50 KB）并在多轮后续对话中保持不变时，缓存命中率提升至 **80–97%**。提升幅度归因于初始注入后上下文 hash 的稳定化，使得后续对话轮次主要可由缓存前缀服务。

### 最佳性能条件

要达到上述效果，需满足以下条件：

1. **对话期间文档不可变。** 注入的长内容不能在会话中途被替换或覆盖。每次文档切换都需要完全重新注入，并使现有缓存失效。
2. **会话内模型一致性。** 单个会话内频繁切换模型可能会部分刷新上下文缓存，降低后续轮次的命中率。
3. **辅助提示组件稳定性。** 自动修改核心文件（如 `MEMORY.md`）的心跳例行程序等后台进程会改变系统提示词 hash，从而导致缓存失效。如果存在此类进程，应在注入窗口期内抑制它们，或配置为仅在用户明确授权时才修改文件。

### 诊断建议

如果注入静态文档后观察到的缓存命中率低于 50%，建议执行以下诊断步骤：

- **验证提示词稳定性。** 确认在测试窗口期内没有后台进程修改系统提示词中引用的文件。
- **检查模型一致性。** 确保在整个注入和后续对话轮次中使用的是同一个模型实例。
- **评估文档大小。** 小于 50 KB 的文档可能无法产生足够大的缓存前缀以产生有意义的命中率提升。

> **注意：** 此处报告的缓存命中率来源于单个 OpenClaw 会话中的经验观察。它们代表参考数据而非统计上严格的数据。正式的 A/B 测试需要独立的会话，使用相同的注入内容和相同的对话协议。

## 📊 观测结果（仅供参考）

本次测试在**单个 OpenClaw 会话**内进行，轮次间切换了模型。第一个模型的上下文残留可能影响了第二个模型的缓存统计。**这不构成严格的 A/B 测试。**

下方数据仅反映该特定会话中观察到的缓存行为，应视为**参考数据**，而非决定性的跨模型对比。严格的 A/B 测试需要两个独立会话，分别注入相同数据并独立执行。

| 模型 | 缓存命中率 | 上下文使用 | 轮次 | 新写入 | 会话隔离 |
|------|-----------|-----------|------|--------|---------|
| **DeepSeek V4 Flash** | **97%** | 56k / 1.0M (5%) | 5 | 0 | ❌ 同会话 |
| **Agnes 2.0 Flash** | **89%** (⚠️ 残留) | 65k / 1.0M (6%) | 5 | 0 | ❌ 同会话 |

**观察：**

- 两个模型在所有轮次中均表现出稳定的上下文缓存，零压缩。
- 两个测试中新 token 写入量均为 0，确认复用模式有效。
- DeepSeek V4 Flash 命中率更高（97% vs 89%），但需要跨会话验证。

## 🛠️ 复现步骤

1. 准备一份长格式研究文档（50KB+，`.md` 格式）
2. 读取并将文档注入 OpenClaw 会话上下文
3. 执行 3-5 轮简短后续对话
4. 通过 `session_status` 检查缓存命中率
5. 切换模型，重复步骤 2-4

> **如需严格的 A/B 测试：** 在两个独立的隔离会话中分别执行步骤 2-4，注入相同的内容并使用相同的对话提示。

## 🔗 相关项目

### Dynamic Token Compression — 实时数据管线

Preload & Cache 与 Dynamic Token Compression 互为补充，构成单一的上下文管理策略的两个方面：

| 维度 | Preload & Cache | Dynamic Token Compression |
|------|----------------|--------------------------|
| **数据类型** | 静态长文档 | 实时动态搜索结果 |
| **使用场景** | 对同一份报告的反复查询 | 对新鲜数据的单次查询 |
| **核心技术** | 上下文缓存 + 注入 | 结构化压缩 + 比率优化 |
| **最佳用途** | 对一份文档 5+ 次查询 | 对抓取数据的单次查询 |
| **参考** | 本文档 | [../dynamic-token-compression/](../dynamic-token-compression/) |

**两者共同构成完整管线：**

```
实时数据 → Dynamic Token Compression → 压缩结果
                                            ↓
                                  若反复使用 → Preload & Cache
```

详见 [Dynamic Token Compression README](../dynamic-token-compression/README_zh-CN.md)，了解压缩比、策略基准和决策规则。
